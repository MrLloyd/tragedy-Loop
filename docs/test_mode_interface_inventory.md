# 惨剧轮回：测试模式接口清单

本文件专门记录 **测试模式 / 调试局** 当前允许调用的接口、边界约束、与正式版的隔离原则。

目的不是描述规则本体，而是为了后续排查时快速判断：

- 这是 **测试模式适配层问题**
- 还是 **正式 engine / phase / resolver 问题**
- 是否存在 **测试模式专用行为** 误伤正式版

配套文档：

- `docs/engine_ui_boundary_mapping.md`
- `docs/rules_to_engine_mapping.md`
- `docs/rules_to_ui_mapping.md`

---

## 1. 硬约束

测试模式后续开发与排查，默认遵守以下硬约束：

- 测试模式只走 **测试模式专用入口 / 适配层**
- 正式版流程只走 **正式开局 / 正式控制器 / 正式 UI**
- 除非明确批准，否则 **不为了测试模式去改正式版行为**
- 每次新增测试模式能力时，都要先记录：
  - 复用了哪些正式逻辑
  - 新增了哪些测试模式专用行为
  - 哪些点需要正式版手测

一句话：

- 测试模式可以复用正式逻辑
- 但测试模式不能和正式流程混成一层

## 2. 当前测试模式入口

当前测试模式主入口：

| 主题 | 文件 |
|------|------|
| 标题页入口 | `ui/screens/title_screen.py` |
| 主窗口切换 | `ui/main_window.py` |
| 测试模式适配层 | `ui/controllers/test_mode_controller.py` |
| 测试模式界面 | `ui/screens/test_mode_screen.py` |

当前推荐的排查顺序：

1. 先看 `ui/screens/test_mode_screen.py`
2. 再看 `ui/controllers/test_mode_controller.py`
3. 再回到 engine 复用点

## 3. 当前允许复用的正式逻辑

测试模式当前允许复用以下正式逻辑：

| 类型 | 接口 / 对象 | 文件 | 说明 |
|------|-------------|------|------|
| 调试建局 | `build_debug_state(...)` | `engine/debug/api.py` | 创建调试局 |
| 调试前置状态 | `apply_debug_setup(...)` | `engine/debug/api.py` | 应用测试模式配置 |
| 调试快照 | `get_debug_snapshot(...)` | `engine/debug/api.py` | 读取测试模式当前快照 |
| 调试能力枚举 | `list_debug_abilities(...)` | `engine/debug/api.py` | 枚举可触发能力 |
| 调试能力触发 | `trigger_debug_ability(...)` | `engine/debug/api.py` | 手动触发能力 |
| 调试事件触发 | `trigger_debug_incident(...)` | `engine/debug/api.py` | 手动触发事件 |
| 阶段处理器注册 | `create_phase_handlers(...)` | `engine/phases/phase_base.py` | 复用正式阶段处理器 |
| 阶段执行 | `PhaseHandler.execute(...)` | `engine/phases/phase_base.py` | 复用正式阶段执行逻辑 |
| 状态机推进 | `StateMachine` | `engine/state_machine.py` | 复用正式阶段推进 |
| 常驻同步 | `settle_persistent_effects(...)` | `engine/rules/persistent_effects.py` | 执行前刷新常驻效果 |

这些接口可以复用，因为它们本身已经是：

- engine 层标准逻辑
- 或调试专用 engine 接口

## 4. 当前测试模式专用适配层

以下逻辑属于测试模式专用，不应反向污染正式版：

| 主题 | 文件 | 说明 |
|------|------|------|
| 测试模式草稿态 | `ui/controllers/test_mode_controller.py` | 保存当前测试配置 |
| 目标下拉整理 | `ui/controllers/test_mode_controller.py` | 将目标候选转成 UI 下拉 |
| 测试模式按钮行为 | `ui/screens/test_mode_screen.py` | 应用 / 触发 / 阶段推进 |
| 连续触发不自动重建 | `ui/screens/test_mode_screen.py` | 测试模式专用行为 |
| 最小版阶段推进 UI | `ui/screens/test_mode_screen.py` | 执行当前阶段 / 推进下一阶段 |
| 等待输入展示 | `ui/screens/test_mode_screen.py` | 当前只展示，不继续提交 |

这部分如果出问题，**优先修测试模式适配层**，不要直接改正式版。

## 5. 当前测试模式能力范围

当前测试模式已经支持：

- 选模组进入调试局
- 配置规则 Y / 规则 X，并重建测试对局
- 自由放角色，不按正式剧本身份池限制
- 修改角色状态
  - 身份
  - 区域
  - token
  - 生死
  - 公开
  - 移除
- 修改当前轮回 / 当前天数 / 当前阶段
- 手动触发事件
- 手动触发身份能力
- 手动触发规则能力
- 最小版阶段推进
  - 执行当前阶段
  - 推进到下一阶段

当前测试模式尚未完整支持：

- 阶段内 `WaitForInput` 的继续提交
- 角色能力专项触发入口

## 6. 当前已确认的测试模式约束

### 6.1 连续触发时不自动重建

当前行为：

- 点“触发事件 / 触发身份能力”时，不自动重建调试局
- 这样可以连续叠加观察现场变化

排查意义：

- 如果连续触发后现场被覆盖，优先查测试模式适配层
- 不要先怀疑正式能力本体互相抵消

### 6.2 修改上方配置后，必须显式应用

当前行为：

- 修改角色、轮回、天数、阶段等输入后
- 需要点 `应用到调试局`
- 否则后续触发仍基于当前已建立的调试局

排查意义：

- 如果“界面值”与“实际结算值”不一致，先检查是否已应用

### 6.3 测试模式固定 3 轮回 / 3 天

当前行为：

- 测试模式统一采用 `3轮回 / 3天`
- 界面只切换当前轮回与当前天数

排查意义：

- 轮回 / 天数超界时，先查测试模式约束，不要先怀疑正式剧本

### 6.4 修改规则 X / Y 会重建测试对局

当前行为：

- 测试模式界面提供规则 Y / 规则 X 选择
- 点击 `应用规则并重建对局` 后，会重新建立测试对局
- 当前现场不会保留

排查意义：

- 规则改完后现场被清空，属于测试模式约定行为
- 如果规则切换后残留旧状态，优先查测试模式重建流程

---

## 7. 规则 X / Y 的后续硬规则

这是后续必须遵守的测试模式约束：

- **修改规则 X / Y 会重新建立测试对局**
- **当前现场不会保留**

这条约束成立的原因：

- 规则 X / Y 会影响常驻效果、派生身份、阶段能力、失败条件
- 在当前对局里热切换，最容易留下旧规则残留状态
- 重建调试局虽然牺牲便利性，但更容易保证状态干净

因此后续如果补测试模式的规则 X / Y 编辑能力，应按下面语义实现：

1. 更改规则 X / Y
2. 重建调试局
3. 当前现场全部丢弃
4. 明确提示用户：这是新的测试对局

排查时不要误判为 bug：

- 改规则后先前 token 不见了
- 改规则后先前死亡 / 公开 / 事件结果丢失
- 改规则后当前等待输入中断

这些都应视为 **测试模式约定行为**，不是正式逻辑异常。

注意：

- 本约束是 **测试模式约束**
- 不是正式版“运行中换规则”的支持承诺

---

## 8. 当前最小版阶段推进的边界

当前最小版阶段推进只保证：

- 复用正式 `PhaseHandler.execute(...)`
- 复用正式 `StateMachine.advance(...)`
- 允许观察阶段执行后的现场变化

当前不保证：

- 阶段内所有 `WaitForInput` 都能在测试模式里继续完成
- 测试模式能完整跑完整局
- 测试模式等同于正式对局 UI

因此排查时要分清：

### 8.1 这些更像正式逻辑问题

- 执行阶段后，应该触发的能力没触发
- 失败条件在错误阶段出现
- 常驻效果 / 派生身份刷新不对
- 阶段推进顺序错误

### 8.2 这些更像测试模式问题

- 按钮点了但没把当前状态显示出来
- 明明进入 `WaitForInput`，测试页没提示
- 连续执行阶段时状态没同步到当前显示

---

## 9. 建议的排查分类

以后凡是测试模式相关改动，建议在说明里固定使用下面三类：

### 9.1 复用原逻辑

用于说明：

- 直接复用了正式 resolver / phase / state machine / 判定逻辑

### 9.2 测试模式专用接口

用于说明：

- 只存在于测试模式，不进入正式版流程的入口与适配

### 9.3 新增 / 绕过正式流程（需重点手测）

用于说明：

- 为了测试便利新增的 UI 行为
- 和正式版不完全一致的路径
- 需要在正式版再验证一次的部分

---

## 10. 文档维护规则

出现以下变化时，必须更新本文件：

- 测试模式新增 engine 复用接口
- 测试模式新增 UI 按钮或操作入口
- 测试模式接入规则 X / Y 编辑
- 测试模式开始支持新的 `WaitForInput`
- 测试模式与正式版的隔离边界变化
- 测试模式出现新的“约定性重建 / 不保留现场”行为

如果以后正式版排查中发现：

- 某问题只出现在测试模式
- 某行为是测试模式专用约定
- 某接口本来只允许测试模式调用

也应立即补进本文件。
